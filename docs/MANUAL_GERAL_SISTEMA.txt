MANUAL GERAL DO SISTEMA NEXUS ONE
=================================

Arquivo: docs/MANUAL_GERAL_SISTEMA.txt
Projeto: Nexus One / Gestao Pro
Objetivo: explicar, em linguagem de estudo, como o sistema foi criado, como o front-end conversa com o back-end e qual a funcao geral de cada entidade/modulo.


1. VISAO GERAL DO SISTEMA
=========================

O Nexus One e um sistema de gestao operacional para empresas. Ele junta em um unico painel:

- Login e controle de acesso.
- Painel principal / visao geral.
- Clientes.
- Produtos, estoque, fornecedores, categorias e marcas.
- Pedidos, PDV, orcamentos, separacao, fiscal e follow-up comercial.
- Caixa.
- Financeiro, contas, recorrencias, cobrancas e conciliacao.
- Colaboradores e permissoes.
- Relatorios.
- Logistica.
- Servicos / ordens de servico.
- Painel de controle administrativo, planos comerciais, filiais, contratos, liberacao de modulos e auditoria.

O sistema tem duas partes principais:

FRONT-END
- Local principal: frontend/src
- Tecnologia: React + Vite.
- Responsabilidade: telas, formularios, tabelas, filtros, dashboards, chamadas para API e experiencia do usuario.

BACK-END
- Local principal: backend/projectoads/projectoads/src/main/java/br/com/diego/projectoads
- Tecnologia: Java + Spring Boot + Spring Security + JPA/Hibernate.
- Banco: PostgreSQL.
- Responsabilidade: regras de negocio, autenticacao, validacoes, persistencia no banco, endpoints REST, fiscal, financeiro, auditoria e integracoes.


2. FLUXO GERAL DE DADOS
=======================

O fluxo mais comum do sistema segue esta ordem:

1. Usuario acessa uma tela no front.
2. A tela chama uma funcao de frontend/src/services/resources.js.
3. resources.js usa frontend/src/services/api.js para fazer HTTP.
4. O back recebe a chamada em um Controller.
5. O Controller chama um Service.
6. O Service aplica regra de negocio.
7. O Service usa um Repository para ler/gravar no PostgreSQL.
8. O back retorna DTO/JSON.
9. O front transforma os dados em viewModels e renderiza componentes.

Exemplo:

Tela Produtos
-> endpoints.produtos.listar()
-> GET /produtos
-> ProdutoController
-> ProdutoService
-> ProdutoRepository
-> tabela/model Produto
-> retorna lista de produtos
-> Produtos.jsx monta a tela com componentes e viewModels.


3. ESTRUTURA DO FRONT-END
=========================

Local: frontend/src

Arquivos raiz:

- main.jsx
  Ponto de entrada do React. Monta o app no navegador.

- App.jsx
  Componente principal. Depois da separacao, ficou mais enxuto e delega rotas/layouts.

Pastas principais:

- routes/
  Define as rotas da aplicacao.
  AppRoutes.jsx organiza as telas.
  PrivateRoute.jsx protege telas que exigem login.

- layouts/
  Define a estrutura visual base.
  AuthLayout.jsx: layout para login/autenticacao.
  DashboardLayout.jsx: layout principal apos login.
  Sidebar.jsx: menu lateral.
  Header.jsx: topo do sistema.

- pages/
  Cada pasta representa uma tela/modulo maior do sistema.
  Exemplo: Login, Dashboard, Clientes, Produtos, Pedidos, Financeiro.

- components/
  Componentes reutilizaveis entre varias telas.
  Exemplo: Button, Modal, Table, StatusBadge, HeaderSearch, HeaderAlerts.

- services/
  Camada de comunicacao com o back-end.
  api.js configura a base HTTP.
  auth.js cuida de token/sessao.
  resources.js centraliza endpoints.

- utils/
  Funcoes auxiliares reutilizaveis.
  Exemplo: formatCurrency, formatDate, permissions, stock, products, exporters.

- constants/
  Constantes compartilhadas.
  Exemplo: roles, modules, status, fiscal, admin.

- styles/
  CSS global separado por responsabilidade.
  Exemplo: base, layout, forms, tables, cards, controls, responsive.

19. CAMADAS DO BACK-END
=======================

controller/
- Recebe HTTP.
- Valida parametros basicos.
- Chama services.
- Retorna ResponseEntity/DTO.

service/
- Concentra regra de negocio.
- Cria, atualiza, cancela, baixa, finaliza, fatura, valida.
- Pode chamar outros services.
- Pode registrar auditoria.

repository/
- Interface com banco via Spring Data JPA.
- Busca por ID, lista, filtros e consultas customizadas.

model/
- Entidades persistidas no banco.
- Representam tabelas e relacoes.

dto/
- Objetos de entrada e saida.
- Evitam expor entidade diretamente em alguns fluxos.

security/
- Spring Security.
- JWT.
- CORS.
- UserDetails.
- Hierarquia de roles.

config/
- Configuracoes gerais.
- Conversores.
- Bootstrap.
- Auditoria de delete.
- Enums.

exception/
- BusinessException.
- GlobalExceptionHandler.
- Padroniza erros.

23. PONTOS DE ATENCAO PARA MANUTENCAO
=====================================

- Manter nomes internos de propriedades sem acento quando ja fazem parte da API.
  Exemplo: usuario, recorrencia, garantiaAte, pecas.

- Usar acento em textos visiveis na tela.
  Exemplo: "Próximo", "Não", "Peças", "Evidências".

- Evitar logica pesada dentro de componentes JSX.
  Preferir viewModels, hooks e services.

- Ao criar um modulo novo:
  1. Criar page.
  2. Criar components.
  3. Criar hooks se houver estado/acoes.
  4. Criar viewModels se houver transformacao de dados.
  5. Registrar endpoint em resources.js.
  6. Criar controller/service/repository/model no back se precisar persistir.

- Ao mexer em endpoints:
  1. Conferir resources.js.
  2. Conferir Controller correspondente.
  3. Rodar build do front.
  4. Rodar testes/compile do back quando possivel.

- Ao mexer em permissao:
  1. Conferir constants/modules.js e roles.js.
  2. Conferir utils/permissions.js.
  3. Conferir Usuario/Permissao no back.
  4. Conferir liberacao de modulo por empresa.

25. RESUMO FINAL
================

O Nexus One foi organizado como uma aplicacao de gestao modular.

No front, cada modulo deve ficar separado em:
- Page principal.
- Components.
- Hooks.
- ViewModels.
- Services quando necessario.
- CSS proprio quando fizer sentido.

No back, cada area segue o padrao:
- Model representa a tabela.
- Repository conversa com o banco.
- Service aplica regra de negocio.
- Controller expoe endpoint REST.
- DTO transporta dados de entrada/saida.

O ponto mais importante para entender o projeto e sempre seguir o fluxo:

Tela -> resources.js -> Controller -> Service -> Repository -> Model/Banco

Esse caminho permite estudar qualquer funcionalidade do sistema sem se perder.
